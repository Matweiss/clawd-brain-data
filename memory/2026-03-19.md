## Browser coworking SOP / anti-relay rule (March 19)
- Mat explicitly wants the assistant to use the **real visible Mac Chrome coworking browser via Chrome DevTools / MCP-style remote debugging**, not the OpenClaw relay/browserless tabs
- Hard rule going forward: **DO NOT default to the OpenClaw Browser Relay extension path for coworking** when the goal is a real visible browser Mat can see/control
- Earlier confusion came from using `chrome-relay` / relay-side tabs, which are not the same as the actual visible coworking tab/window Mat expects
- Correct target is Mat's real Chrome DevTools endpoint on the Mac (remote debugging browser), which was confirmed working when forwarded to the VPS on a non-conflicting port

### Working transport pattern that succeeded
- Mac Chrome remote debugging was available locally on `127.0.0.1:18800`
- To avoid collision with the VPS OpenClaw headless browser also using `18800`, the successful bridge was:
  - Mac local: `127.0.0.1:18800`
  - VPS forwarded port: `127.0.0.1:28800`
- Successful SSH remote forward command from Mac to VPS:
  - `ssh -N -R 127.0.0.1:28800:127.0.0.1:18800 root@srv882799.hstgr.cloud`
- Verification from VPS succeeded:
  - `http://127.0.0.1:28800/json/version`
  - returned a **Mac Chrome** user agent, not Linux HeadlessChrome
- Real tab creation succeeded by hitting the forwarded Chrome DevTools endpoint directly from VPS:
  - `PUT /json/new?https://...`
- This is the preferred coworking/browser-control path until a cleaner always-on version is established

### Important diagnostic rules
- If forwarded DevTools on VPS shows `HeadlessChrome` / Linux, it is the wrong browser path (likely VPS/browserless collision)
- Never forward the Mac's `18800` to VPS `18800` if the VPS already uses that port for OpenClaw's own browser/CDP service
- Use a non-conflicting forwarded port like `28800`
- If browser relay extension shows `127.0.0.1:18792` unreachable, that is not the desired primary path anyway for Mat's preferred SOP

### Long-term requirement recorded
- Mat wants this cowork/browser-control path available **24/7**, especially so scheduled overnight data pulls can run without manual terminal commands
- Long-term goal: persistent always-on bridge from Mac Chrome DevTools to VPS so the assistant can access the real browser while Mat sleeps
- Desired future state:
  - no manual SSH tunnel startup
  - no relay extension dependency
  - browser access survives restarts / normal operation
  - suitable for scheduled data pulls and live coworking
- This should become a persistent Mac service / LaunchAgent-style tunnel or equivalent always-on bridge, separate from the already-fixed OpenClaw node/tunnel setup
